# 포트폴리오 프로젝트 규칙

## 작업 목표
이력서 기반 콘텐츠 갱신과 코드 개발(버그 수정, 기능 개선 등) 모두 진행한다.
콘텐츠는 "콘텐츠 수정 등급", 코드는 "코드 작업 승인 체계"에 따라 진행한다.


## 제외 항목
- 비공개 스터디 사이트(경제공부/투자 대시보드) 미포함
- IT 직무 전환 멘토링 (2025.08~현재) 미포함
- 프로젝트 기간: 꼭 필요한 경우만 표기, 애매하면 표기 전 사용자에게 확인


## 콘텐츠 수정 등급
- 개인정보/신원 정보(확정 정보·경력·학력·자격증 등 "타인이 나를 이해하는 데 필요한 정보"): 보수적. 수정 전 반드시 사용자 확인
- 그 외 문구(메뉴바 라벨, 섹션 제목, 소개 카피 등): 추천·자율 수정 가능. 단 방어선 규칙은 항상 적용되고, 수정 후 곧바로 커밋하지 않고 사용자가 검토·재수정할 수 있게 둘 것

## 기술 구조 참고
- Next.js 15 (App Router, turbopack) / React 19 / Tailwind
- app/api/sendMessage: 챗봇 응답 처리 (OpenAI Vector Store 기반 RAG + MongoDB 로그 + Telegram 알림) — 아래 "코드 작업 승인 체계" 1번 적용 대상
- .env.local의 API 키·DB 접속정보(OPENAI_API_KEY, MONGO_URI 등)는 코드·커밋·응답 어디에도 노출 금지

## local-notes/ 폴더
- git에 올라가지 않는 개인 작업 공간 (.gitignore 등록됨)
- 포트폴리오 사이트와 직접 관련 없는 파일 보관용: 작업 기록(오늘 한 일), 다음에 할 일 목록, 이력서 원본 등
- 세션 중 이런 성격의 메모·요약이 필요하면 이 폴더에 저장

## 세션 작업 규칙
- 매 세션 종료 시, local-notes/diary/ 폴더에 날짜 파일
  (YYYY-MM-DD.md)로 작업 일기를 남긴다. 폴더가 없으면 만든다.
  같은 날 여러 세션이면 파일에 이어 붙인다.
  일기 내용: 이번 세션에 한 일 / 바꾼 파일 / 검증 결과. 간단하게.

## 코드 작성 원칙
- 확실하지 않으면 멋대로 해석해서 진행하지 말고 먼저 질문
- 요청 범위보다 코드를 부풀리지 않기 (3줄이면 될 걸 불필요한 설정·추상화로 키우지 않기)
- 요청과 무관한 주변 코드(포매팅·주석·다른 함수)는 건드리지 않기

## 코드 작업 승인 체계
1. app/api/* (챗봇 로직: OpenAI·MongoDB·Telegram 연동) — 완전 보수적
   - 임의 진행 금지. 충분한 대화로 범위를 합의한 뒤에만 진행
   - 합의된 범위만 정확히 수정 (관련돼 보이는 부수 변경도 임의로 하지 않고 별도 보고)
2. package.json 의존성 추가/삭제 — 승인 게이트
   - 왜 필요한지 설명 → 사용자가 이해했는지 확인 → 허락받고 진행
3. 그 외 일반 코드 변경 (컴포넌트, 스타일, 버그 수정 등)
   - 비전공자 눈높이로 쉽게 설명 → 이해 확인 → 허락받고 진행

## 완료 기준
테스트 스위트 없음(package.json에 test 스크립트 없음). "됐다"고 말하기 전에:
- UI/콘텐츠 변경 → 먼저 dev 서버(preview 도구)로 직접 확인 후 전달, 최종 확인은 사용자가 진행
- app/api/* 변경 → 실제 요청을 보내 응답 확인
- 설정/빌드 변경 → npm run build, npm run lint 통과 확인
- 확인 결과가 실패하면 원인을 고치고 다시 확인 → 통과할 때까지 반복. 통과 후 무엇을 어떻게 확인했는지 결과와 함께 보고

## git 정책
- commit: 작업의 주요 지점마다 자율적으로 진행 가능 (매번 묻지 않음). 단 "그 외 문구" 자율 수정분은 사용자 검토 전 커밋 보류
- push: 항상 사용자 동의 필수 (예외 없음) — push 시 Vercel 자동 배포로 실사이트(kysportfolio.site)에 반영될 수 있음
- 커밋 메시지: 한국어로, 최대 2줄 이내로 이해하기 쉽게 작성

## 절대 금지
- .env.local 파일 직접 수정
- MongoDB 데이터 직접 삭제·변경 (스크립트를 통한 프로덕션 DB 직접 접근)
- git push --force, git reset --hard
- API 키·DB 접속정보를 코드·커밋·채팅 응답에 노출
